Get started

Build and provision the stack

The repository contains two pinned PlatformIO projects, a Python archiver, a static browser app and a deterministic system simulator.

Field trials, not production. LoRa frames and BLE provisioning are not cryptographically authenticated. Keep the system private and read the production readiness assessment.

Requirements

  • PlatformIO Core or the VS Code extension
  • Python 3.11 or newer
  • Node.js 20 or newer for browser tests
  • A private MQTT broker with trusted TLS TCP and WebSocket listeners

Provision unique secrets

Copy each firmware component’s secrets.example.h to its git-ignored secrets.h. Give every physical device a different 12+ character onboarding password and a separate OTA password hash. The gateway also needs the broker’s PEM root CA. Leave plaintext MQTT disabled.

Compile the trackers

pio run -d components/tracker-firmware \
  -e heltec_wifi_lora_32_v2

pio run -d components/tracker-firmware \
  -e heltec_wireless_tracker

Choose only the environment matching the exact board. On first boot, join LoRaTracker-<device_id>, then authenticate to HTTP as admin with the unique onboarding password. The setup ID is derived from the eFuse MAC until you assign the final identity.

Compile the gateway

pio run -d components/gateway-firmware \
  -e heltec_wifi_lora_32_v2

The checked-in gateway pin map targets only Heltec WiFi LoRa 32 V2/SX1276. Register every allowed tracker; first boot has an empty allowlist. A provisioned gateway requires both HTTP authentication and a five-second USER-button hold for mutations.

Run the services

cd components/archiver
python -m venv .venv
. .venv/bin/activate
pip install .
cp .env.example .env
python -m lora_tracker_archiver

# In another terminal
cd components/web-app
python3 -m http.server 8080

Serve the web app with HTTPS and connect it to wss:// outside local development.

Acceptance sequence

  1. Run Python, browser, simulator and all three PlatformIO builds.
  2. Verify the gateway’s trusted TLS connection and broker ACLs.
  3. Verify matching radio settings and a registered tracker hash.
  4. Confirm plausible GNSS time, position, battery and increasing sequence.
  5. Confirm ACK-driven queue progress and a history response ending in final=true.
  6. Record hardware revision, firmware commit and configuration revision.

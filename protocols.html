Strict contracts between components

Only the current packet, point and history-request schemas are supported. Incompatible input is rejected instead of silently translated.

Identity

A canonical lowercase device ID is hashed into a public 64-bit routing identifier. The stable point identity is:

<device_hash>:<boot_id>:<sequence>

The archiver verifies that all three parts agree with the payload. The routing hash is public and is never an authentication credential.

LoRa frames

The packed little-endian envelope identifies transport, message type, schema, flags and device hash. History uses an absolute coordinate plus signed spatial deltas, one absolute GNSS UTC value and ULEB128 time deltas. The gateway rejects unknown devices, unsupported schemas, malformed deltas and inconsistent timestamps.

MQTT event model

lora-tracker/v1/trackers/<hash>/events/point
lora-tracker/v1/trackers/<hash>/state
lora-tracker/v1/trackers/<hash>/history/request
lora-tracker/v1/trackers/<hash>/history/response/<request>

Point events are non-retained. Current state, availability and status are retained. Gateway TCP MQTT and browser WebSockets should use trusted TLS; the archiver and gateway fail closed unless plaintext is explicitly enabled for testing.

Configuration and onboarding

Tracker and gateway settings use CRC-protected, revisioned blobs with active and backup NVS copies. HTTP uses Basic authentication and transactional expected_revision patches. Gateway writes also require a physical unlock. BLE shares the transaction model but remains unauthenticated and is a release blocker.
